# @servicebay/frontend

The UI surface for ServiceBay — components, hooks, dashboards, providers, the install wizard. Phase 3 of [#753](https://github.com/mdopp/servicebay/issues/753) moved this into its own workspace package; Phase 4 (this PR onward) lets a frontend-only contributor run the UI in dev mode **without the real backend running**.

## Run the frontend in isolation

```bash
# From the repo root:
npm run dev:frontend
```

That starts Next.js on **port 3001** with `NEXT_PUBLIC_USE_MOCKS=1` set. A [Mock Service Worker](https://mswjs.io/) intercepts every `fetch('/api/...')` and returns a fixture-shaped response that satisfies the same zod schemas the real backend uses. No SSH agent, no backend services, no podman.

The full integrated dev path is still `npm run dev` (port 3000, custom server in `server.ts`, all the way down to a real agent socket). Use that when you need a real backend; use `dev:frontend` when you're iterating on the UI.

## Layout

```
packages/frontend/
├── public/
│   └── mockServiceWorker.js     # Generated by `npx msw init`. Don't edit by hand.
├── src/
│   ├── components/              # React components, including wizard
│   ├── dashboards/              # Top-level pages (Network, Services, Health, …)
│   ├── hooks/                   # Custom hooks
│   ├── mocks/                   # MSW layer — fixtures + handlers + worker setup
│   │   ├── browser.ts           # setupWorker(...handlers)
│   │   ├── fixtures.ts          # Schema-valid sample responses
│   │   ├── handlers.ts          # http.* handlers for each endpoint
│   │   └── init.ts              # Conditional boot, guarded by NEXT_PUBLIC_USE_MOCKS
│   └── providers/
│       └── MockProvider.tsx     # Client component that boots MSW before rendering
└── package.json
```

## Files NOT to edit from a frontend-only PR

These are owned by the backend dev. Changes to them go through the backend test suite + integration tests, not the frontend dev loop:

- `packages/backend/**` — agent, install, diagnose, network, store.
- `packages/api-client/**` — the typed contract. Both sides depend on these schemas; widening or narrowing one is a backend-author decision.
- `src/app/api/**` — Next.js route handlers. Hand-rolled validation lives here.

If a frontend feature needs a new endpoint or a wider response shape, **the schema in `@servicebay/api-client` is the place to start the conversation** — that change is what unlocks both the mock handler in this package and the real route handler in `src/app/api/`.

## Adding a new mocked endpoint

1. **Schema first.** Add (or widen) the `…RequestSchema` / `…ResponseSchema` in the relevant `packages/api-client/src/*.ts`.
2. **Fixture.** Add a happy-path fixture in `src/mocks/fixtures.ts`. Export a `bad-path` variant only when a story / Storybook variant genuinely needs to exercise the error path.
3. **Handler.** Register an `http.*(...)` handler in `src/mocks/handlers.ts`. Always parse the request body with the schema so a request shape mismatch surfaces here rather than at runtime against the real backend.
4. **README.** If the endpoint is part of the "runs without a backend" contract — e.g. it's hit during initial wizard render — add it to the list below.

Currently mocked endpoints (Phase 2):

| Endpoint | Method | Schema | Notes |
|---|---|---|---|
| `/api/services/validate-yaml` | POST | `ValidateYamlRequest/Response` | `?mock=invalid` for the parse-error fixture |
| `/api/install/generate-secret` | POST | `GenerateSecretRequest/Response` | Deterministic secret string |
| `/api/templates/parse-dependencies` | POST | `ParseDependenciesRequest/Response` | Returns `['auth', 'nginx']` |
| `/api/install/status` | GET | `InstallStatusResponse` | Idle, no active job |

Other endpoints fall through to the real network — set `onUnhandledRequest: 'bypass'` in `init.ts` so an unmocked endpoint produces a real 404/500 instead of a console warning. If you find yourself hitting a lot of unmocked endpoints while iterating on a screen, that's a signal to widen this handler list.

## Note on Storybook

A Storybook integration was scoped, partially landed (PR #790 in 4.13.0), and then **rolled back** — the cost-benefit didn't make sense for the team shape (solo dev). Storybook's main value is team collaboration (designers without backend, FE hires without onboarding); the bulk of the "iterate on UI without the backend" value is already covered by `npm run dev:frontend` above. The webpack stubbing required to keep backend code out of the browser bundle was fragile against Storybook + Next 16 + the api-client's `export type * from '@/lib/...'` re-export chain, and rendering bugs kept slipping past the build.

If a future team scale brings a FE-focused contributor, Storybook can be re-introduced; in the meantime, the mock-layer foundation in `src/mocks/` is the supported FE-only dev path.
